# 设计规范 — SwissArmyPM

最后更新：2026-04-11

---

## 一、全局设计原则

### 1.1 产品定位

SwissArmyPM 是**只服务项目经理**的桌面工作台。所有设计决策以此为中心。

### 1.2 核心设计原则

| 原则 | 含义 | 实践指引 |
|------|------|----------|
| **快速录入** | 项目经理的日常是频繁更新，不是悠闲浏览 | 所有表单默认 inline 编辑；减少弹窗层级；支持键盘快捷键 |
| **信息密度** | 项目经理需要在有限屏幕面积内同时掌握多个维度 | 合理使用表格和紧凑布局；允许折叠/展开次要信息；关键指标常驻 |
| **证据可追溯** | 项目事实需要能回溯到来源 | 所有结构化数据支持"来源"字段；证据链接可点击跳转；变更历史可查 |
| **手工优先** | 第一阶段不强依赖 AI 自动化 | CRUD 操作必须完整可用；不依赖自动填充才能完成任务 |
| **桌面独立** | 单机即可完成所有核心操作 | 不假设网络可用；数据全部存本地 SQLite；无云依赖 |

### 1.3 视觉风格

- 主题：亮色默认，暗色可选（Ant Design ConfigProvider + Tailwind dark mode）
- 色彩：专业、克制，以 Ant Design 默认色板为基础
- 字号：基础 14px，表格 13px，辅助信息 12px
- 间距：紧凑但不拥挤，使用 Ant Design 默认 spacing token

## 二、Ant Design 使用规范

### 2.1 组件选择

- 表格数据 → `Table`（带行选择、排序、筛选）
- 表单录入 → `Form` + `Input` / `Select` / `DatePicker`
- 确认操作 → `Popconfirm`（轻量）或 `Modal.confirm`（重大操作）
- 信息展示 → `Descriptions` / `Statistic` / `Tag`
- 导航 → `Menu`（侧边栏）+ `Tabs`（页面内切换）
- 反馈 → `message`（轻提示） / `notification`（重要通知）

### 2.2 一致性要求

- 所有页面使用统一的 `ConfigProvider` 主题配置（`src/renderer/config/theme.ts`）
- 颜色、圆角、间距统一走 Ant Design token，不随意硬编码
- 图标统一使用 `@ant-design/icons`，补充使用 `lucide-react`

## 三、PM Workspace 核心模块设计指导

### 3.1 项目画布（Project Canvas）

**定位**：项目的"一页纸总览"，让项目经理一眼看清项目全貌。

**设计要点**：
- 使用 `Descriptions` 或卡片网格布局，分区展示项目关键属性
- 分区建议：基本信息 / 目标与范围 / 关键里程碑 / 核心干系人 / 当前状态 / 备注
- 支持 inline 编辑（点击即编辑），减少弹窗
- 画布内容变更需要记录变更日志
- 画布支持"来源"标注（某条信息来自哪个证据）

**组件参考**：`Descriptions` + `Typography.Text`（editable）+ `Tag`（状态标签）

### 3.2 干系人（Stakeholders）

**定位**：管理项目所有干系人的关键信息与影响分析。

**设计要点**：
- 使用 `Table` 展示干系人列表，列：姓名 / 角色 / 影响力 / 态度 / 联系方式 / 备注
- 新增/编辑使用 `Drawer`（右侧滑出），不使用弹窗，保持上下文可见
- 支持按影响力-态度矩阵视图切换（权力-利益方格）
- 支持分组/筛选（按角色、按项目阶段）
- 每个干系人条目支持挂接证据来源

**组件参考**：`Table` + `Drawer` + `Select` + `Tag` + 自定义矩阵图

### 3.3 时间规划表（Timeline）

**定位**：项目关键时间节点的可视化与维护。

**设计要点**：
- 默认使用表格视图（类似 Excel 甘特），当前已有 `ExcelGanttChart` 可复用
- 支持甘特图视图切换（当前已有 `VisTimelineWrapper` / `GanttChart`）
- 关键信息：任务名 / 开始日期 / 结束日期 / 负责人 / 状态 / 依赖关系
- 支持拖拽调整时间
- 里程碑用醒目标记（如菱形图标 + 高亮色）
- 时间规划条目支持挂接证据

**组件参考**：现有甘特组件群（`ExcelGanttChart`、`GanttChart`、`VisTimelineWrapper`）

### 3.4 风险登记册（Risk Register）

**定位**：集中管理项目风险，支持快速评估和跟踪。

**设计要点**：
- 使用 `Table` 展示风险列表，列：风险描述 / 类别 / 概率 / 影响 / 风险等级 / 应对策略 / 责任人 / 状态
- 支持概率-影响矩阵热力图视图
- 风险等级自动计算（概率 × 影响），用颜色 Tag 标示（红/黄/绿）
- 新增/编辑使用 `Drawer` 或 inline 编辑
- 风险条目支持关联工作包和证据

**组件参考**：`Table` + `Tag` + `Drawer` + 自定义热力图

### 3.5 工作包（Work Packages）

**定位**：项目的最小可管理工作单元。

**设计要点**：
- 使用 `Table` 展示工作包列表，列：名称 / 描述 / 负责人 / 状态 / 开始 / 结束 / 关联里程碑
- 支持状态流转看板视图（待开始 → 进行中 → 已完成）
- 工作包可关联到时间规划条目和风险
- 支持批量状态更新
- 工作包支持挂接证据

**组件参考**：`Table` + `Tag` + `Select` + 看板布局（`Steps` 或自定义）

## 五、响应式与可用性

- 目标屏幕：1366×768 以上（笔记本到台式机）
- 最小支持：1280×720
- 不需要移动端适配（桌面工具）
- 侧边栏支持折叠
- 表格支持列宽调整和列排序
- 所有表单输入支持 Tab 键顺序切换

### 5.1 桌面端布局的核心原则

SwissArmyPM 是桌面工作台，不采用“窗口一缩小就自动变成移动端单列流式页面”的策略。

必须遵守：
- **不允许把核心工作区自动挤压成信息失真的长列表**
- **不允许所有列同时等比例压缩**，必须有主次优先级
- **不允许通过无限增高卡片去换取窄宽度适配**
- **支持桌面尺寸变化，但不是移动端自适应**

### 5.2 工作台布局优先级规则

以 Project Workspace 为例，布局优先级必须固定为：

1. **中间主工作区优先保宽**
2. 左侧导航次之
3. 右侧 Inspector 最先折叠/抽离

这意味着：
- 当窗口变窄时，不能先压缩 Canvas / Timeline / Risk 主区
- **首选策略是保留语义结构，并通过横向滚动保护主区宽度**
- 应先收缩或折叠右侧 Inspector
- 再考虑左侧 rail 从宽导航变成窄导航/图标导航
- 不允许为了消除滚动而把主区强行改造成完全不同的单列布局

### 5.3 推荐断点不是“移动端断点”，而是“桌面工作模式断点”

建议统一采用以下模式：

#### Full Workspace（≥ 1600）
- 三栏：左导航 + 中主区 + 右 Inspector
- Canvas 保持完整 4 列画布
- 适合全量并行查看

#### Standard Desktop（1366 - 1599）
- 仍保留三栏
- 左 rail 可略缩窄
- 右 Inspector 保持窄列
- Canvas 仍保持 4 列，不降为单列

#### Compact Desktop（1280 - 1365）
- 左 rail 保留
- 中主区优先保宽
- **优先采用主工作区横向滚动，而不是主区变形**
- **右 Inspector 自动切换为 Drawer，不再常驻占据第三列**
- 主区顶部必须提供明确的“打开上下文面板”入口，避免上下文信息消失
- Canvas 允许从 4 列切为 3 列，但**不能退化为单列长列表**

#### Below Minimum Support（< 1280）
- 不做完整布局保证
- 应明确进入“最小宽度以下”状态：
  - 主工作区横向滚动（首选）
  - 或提示用户当前窗口过窄
- 不允许静默退化成完全不同的信息结构

### 5.4 Canvas 模块专用规范

Project Canvas 是 PMBOK 风格画布，不是普通卡片列表。

因此必须遵守：
- 在支持范围内（≥1280）**保持画布语义**，不能自动变成一张张纵向卡片列表
- 画布优先使用固定区域/固定列数逻辑，而不是纯流式 auto-fit
- 内容块的变化应该优先通过：
  - 列数切换（4列 -> 3列）
  - Inspector 抽离
  - 局部滚动
  来处理
- **不通过把 block 高度无限拉长来“适配”窄窗口**

Canvas 建议：
- Full/Standard：4列固定画布
- Compact：优先保留 4 列；必要时降为 3 列，但仍保持画布结构
- <1280：画布容器保留最小宽度，优先通过横向滚动保结构

### 5.5 防止卡片被拉得过高的规范

所有工作台卡片/块必须定义下面三件事：

1. **最小宽度 / 最小列宽**
- 不能无限压窄
- 宽度不足时切换模式，而不是继续挤

2. **最小高度 + 合理上限**
- 例如画布块应有最小高度
- 内容过多时优先内部滚动、摘要折叠、查看更多
- 不能让单个块因为几行文本把整列撑成非常长的瀑布流

3. **文本密度控制**
- 长文本默认摘要、截断或限定展示行数
- 块内正文优先 3~6 行摘要
- 详细内容进入编辑态 / Drawer / Inspector

### 5.6 结构性解决方案，优先级高于样式修补

遇到窗口缩放问题时，优先按以下顺序解决：

1. 调整信息架构（谁主谁次）
2. 调整布局模式（三栏/双栏/抽屉）
3. 调整模块显示策略（摘要/折叠/滚动）
4. 最后才是调字体、padding、gap

禁止把问题只理解成：
- 字体缩小一点
- padding 缩一点
- gap 缩一点

如果结构不对，样式微调只会把问题隐藏，不会解决。

### 5.7 组件规范：谁可以缩，谁不能缩

- 主工作区容器：`min-w-0`，但内容语义不可丢
- 右侧 Inspector：允许折叠 / Drawer 化 / Tab 化
- 左侧导航：允许折叠，但导航语义必须保留
- Canvas block：不可自由退化成 list item
- 数据表格：优先横向滚动，不强行压列
- 表单区：优先双列转单列，但字段分组关系不能被打散

### 5.8 验收标准（必须人工检查）

每个核心页面都要至少检查这几个宽度：
- 1600+
- 1440
- 1366
- 1280

在这些宽度下必须确认：
- 主区是否仍然是主区
- Canvas 是否仍然是画布，不是长列表
- 右侧信息是否已合理抽离，而不是硬挤
- 卡片是否出现异常增高
- 文本是否出现大面积换行导致布局失衡
- 用户是否仍能快速进入“查看 -> 编辑 -> 保存”路径
