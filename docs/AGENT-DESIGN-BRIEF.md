## 2. 产品一句话

SwissArmyPM 是一个**只服务项目经理**的桌面工作台，用于快速维护、查看、校正和追溯项目关键事实。

---

## 3. 绝对不能跑偏的点

- 不做通用协作平台
- 不做 PMO/Portfolio 总览为中心的产品
- 不做 My Work / Pomodoro / Todo 中心
- 不从 0 发明新的通用 agent runtime
- 不让 agent 无声改写核心项目事实

---

## 4. 当前产品中心

当前一等对象只有这几个：

- `Project`
- `ProjectCanvas`
- `Stakeholder`
- `TimelineItem`
- `Risk`

- `FactAssertion`

## 5. 设计时必须遵守的原则

### 5.1 PMBOK 设计原则

- 责任感地管理干系人
- 主动推进关键事项
- 聚焦项目价值
- 持续整合项目真相

### 5.2 结构化优先

先把项目核心对象做好，再叠加 AI 能力。

### 5.3 证据驱动

Evidence 不是附件仓库，而是后续查询、建议更新、来源回溯的基础层。

### 5.4 可追溯优先

重要回答、建议和判断，应尽量返回：

- 结论
- 来源
- 不确定性说明

### 5.5 PM 最终确认

Agent 可以：

- 抽取
- 比对
- 提醒
- 建议

但不能绕过 PM 直接修改核心事实。

---

## 6. 从 GBrain 借鉴什么

只借鉴这些：

- 证据先沉淀，再参与问答和建议更新
- 导入去重、幂等、可审计
- 结构化检索 + 关键词检索 + 语义检索
- 后台持续整理证据，生成候选事实
- 查询时尽量给来源

不要把 SwissArmyPM 设计成泛个人知识大脑。

---

## 7. 输出方案时的偏好

当 agent 产出设计方案时，应优先：

- 给出最小可落地版本
- 明确 P0 / P1 / P2
- 优先支持快速录入、快速更新、快速查看
- 避免为了完整而设计过重流程
- 明确哪些字段/模块直接服务项目经理日常判断
- 明确哪些内容来自证据，哪些内容是结构化事实

---

## 8. 推荐回答方式

如果你是参与设计的 agent，输出时尽量采用这种逻辑：

1. 先说明方案是否符合当前产品中心
2. 再说明它服务的核心对象
3. 再说明它如何满足 PMBOK 设计原则
4. 再说明是否支持 Evidence / Traceability
5. 最后给出最小实现路径
