# Claude Code Master Instructions

## 🎯 Project Overview
**SwissArmyPM** - A lightweight desktop project management tool with AI integration.

## 📁 Directory Structure
```
SwissArmyPM/
├── docs/                  # Documentation
├── design/               # Design system (NOT code)
├── src/
│   ├── main/            # Electron main process
│   ├── renderer/        # React frontend
│   │   ├── ui/        # ═══ FIGMA MAKE OUTPUT ═══ (DO NOT EDIT MANUALLY)
│   │   ├── features/   # ═══ LOGIC + INTEGRATION ═══ (Claude Code works here)
│   │   ├── stores/    # State management
│   │   ├── hooks/     # Shared hooks
│   │   └── pages/     # Route entry points
│   └── shared/        # Shared between main & renderer
└── templates/          # App templates
```

## 🔑 Key Principles

### 1. **UI Layer (Figma Make Output)**
- **Location**: `src/renderer/ui/`
- **Source**: Generated by Figma Make plugin
- **Rule**: DO NOT manually edit these files
- **Updates**: Regenerate from Figma when design changes

### 2. **Feature Layer (Logic + Integration)**
- **Location**: `src/renderer/features/`
- **Purpose**: Connect Figma UI to business logic
- **Claude Code**: This is where you work
- **Pattern**: Container components + custom hooks

### 3. **Main Process (Electron)**
- **Location**: `src/main/`
- **Components**:
  - Database operations (SQLite)
  - File system access
  - AI service integration
  - IPC handlers

## 🛠️ Development Workflow

### When Adding New Features:
1. **UI Components**: Get from Figma Make → `src/renderer/ui/`
2. **Business Logic**: Create in `src/renderer/features/[feature]/`
3. **Integration**: Connect UI to logic via Container components
4. **State**: Use Zustand stores for complex state

### When Modifying Existing Features:
1. Check if changes are UI-related → Update Figma, regenerate
2. Check if changes are logic-related → Modify in `features/` folder
3. Always preserve the separation between UI and logic

### File Naming Conventions:
- **Containers**: `FeatureContainer.tsx`
- **Hooks**: `useFeature.ts`
- **Actions**: `featureActions.ts`
- **Screens**: `FeatureScreen.tsx` (UI layer)
- **Pages**: `Feature.tsx` (Route entry)

## 🚨 Important Rules

1. **Never manually edit** files in `src/renderer/ui/`
2. **Always work** in `src/renderer/features/` for business logic
3. **Consult PRDs** before implementing features
4. **Follow existing patterns** in similar features
5. **Test thoroughly** before committing
6. **Update documentation** when architecture changes
7. **Clarify requirements before coding**: when a user asks for changes, first confirm intent, scope, and interactions (use sketches, step-by-step flows, or examples). Do **not** modify code until both sides agree on the exact behavior.
8. **Avoid rework**: if requirements are uncertain or evolving, pause and align on intent and acceptance criteria before implementing. Prioritize clarity over speed to minimize rework.

## 🤝 Collaboration Notes

- **Figma Make**: UI components and screens
- **Claude Code**: Logic integration and feature implementation
- **Separation of Concerns**: Maintain clear boundary between UI and logic
